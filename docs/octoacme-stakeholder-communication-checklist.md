# OctoAcme — Stakeholder Communication Checklist

## Purpose
Ensure consistent, timely, and effective communication with all project stakeholders throughout the project lifecycle.

## Stakeholder Identification

- [ ] Identify all project stakeholders (internal and external)
- [ ] Categorize stakeholders by role and interest level
- [ ] Document stakeholder contact information and preferred communication channels
- [ ] Determine each stakeholder's information needs and communication frequency
- [ ] Identify key decision makers and approval authorities
- [ ] Map stakeholder relationships and influence

## Communication Planning

- [ ] Create stakeholder communication matrix (who, what, when, how)
- [ ] Define communication objectives for each stakeholder group
- [ ] Establish communication channels (email, Slack, meetings, reports)
- [ ] Set up regular meeting cadence (daily standups, weekly syncs, monthly reviews)
- [ ] Define escalation communication procedures
- [ ] Create templates for common communications (status updates, change notices)
- [ ] Establish response time expectations

## Project Initiation Communications

- [ ] Send project kickoff announcement to all stakeholders
- [ ] Distribute project charter and high-level timeline
- [ ] Share team roster and contact information
- [ ] Communicate project goals and success metrics
- [ ] Clarify stakeholder roles and involvement expectations
- [ ] Schedule kickoff meeting with key stakeholders
- [ ] Provide access to project documentation and collaboration tools

## Ongoing Status Communications

### Weekly Status Updates (Project Manager → All Stakeholders)
- [ ] Progress against milestones this week
- [ ] Completed deliverables and achievements
- [ ] Planned activities for next week
- [ ] Current risks and issues
- [ ] Blockers requiring stakeholder attention
- [ ] Decisions needed from stakeholders
- [ ] Updated timeline or scope changes

### Sprint/Iteration Communications (Product Owner → Stakeholders)
- [ ] Sprint goals and priorities
- [ ] Demo/review invitation and agenda
- [ ] Sprint retrospective outcomes (as appropriate)
- [ ] Backlog updates and reprioritization
- [ ] Feedback requests on delivered features

### Release Communications (Release Manager → All Stakeholders)
- [ ] Release schedule and deployment window
- [ ] Pre-release readiness checklist status
- [ ] Go/no-go meeting invitation
- [ ] Release notes and feature summary
- [ ] Post-deployment verification results
- [ ] Known issues and workarounds

## Risk and Issue Communications

### Risk Identification and Updates (Project Manager → Affected Stakeholders)
- [ ] New risk identified with impact assessment
- [ ] Risk severity level (High/Medium/Low)
- [ ] Mitigation plan and owner assigned
- [ ] Status updates on existing risks
- [ ] Risk closure notifications

### Issue Escalation (Any Role → Project Manager → Stakeholders)
- [ ] Issue description and impact
- [ ] Affected stakeholders and timeline
- [ ] Actions taken so far
- [ ] Requested support or decisions
- [ ] Expected resolution timeline
- [ ] Follow-up communication plan

### Incident Communications (Release Manager/Change Control Coordinator → All)
- [ ] Immediate incident notification (severity, impact, status)
- [ ] Regular status updates during incident response
- [ ] Incident resolution announcement
- [ ] Post-incident review summary
- [ ] Action items and preventive measures

## Change Management Communications

### Change Request Notifications (Change Control Coordinator → Stakeholders)
- [ ] Change request submission acknowledgment
- [ ] Change impact assessment results
- [ ] Change Advisory Board (CAB) meeting invitation
- [ ] Change approval/rejection decision
- [ ] Change implementation schedule
- [ ] Post-implementation review results

### Scope Change Communications (Product Owner/Project Manager → All)
- [ ] Scope change proposal with justification
- [ ] Impact on timeline, budget, and resources
- [ ] Stakeholder feedback and approval request
- [ ] Approved change notification
- [ ] Updated project plan distribution

## Decision Communications

### Decision Documentation (Project Manager → Relevant Stakeholders)
- [ ] Decision context and options considered
- [ ] Decision maker and date
- [ ] Rationale for decision
- [ ] Impact on project plan
- [ ] Action items from decision
- [ ] Who needs to be informed of decision

## Milestone and Phase Transition Communications

### Phase Completion (Project Manager → All Stakeholders)
- [ ] Phase completion announcement
- [ ] Key deliverables and outcomes
- [ ] Lessons learned summary
- [ ] Next phase overview and timeline
- [ ] Updated project status

### Major Milestone Achievement (Product Owner/Project Manager → All)
- [ ] Milestone achievement announcement
- [ ] Impact and business value delivered
- [ ] Recognition of team contributions
- [ ] Next milestones preview

## Project Closure Communications

- [ ] Project completion announcement
- [ ] Final deliverables and outcomes summary
- [ ] Success metrics and results achieved
- [ ] Lessons learned and recommendations
- [ ] Transition plan for ongoing support (if applicable)
- [ ] Thank you and recognition to stakeholders and team
- [ ] Archive location for project documentation

## Communication Best Practices

### Content Guidelines
- [ ] Use clear, concise language appropriate for audience
- [ ] Lead with most important information (executive summary)
- [ ] Use visual aids (charts, dashboards) when helpful
- [ ] Highlight action items and decisions needed
- [ ] Include relevant links to detailed documentation
- [ ] Proofread for clarity and accuracy

### Timing Guidelines
- [ ] Send routine updates on consistent schedule
- [ ] Communicate changes as soon as they are confirmed
- [ ] Provide advance notice for meetings and decisions
- [ ] Escalate critical issues immediately
- [ ] Follow up on action items and commitments

### Channel Selection
- [ ] Email: Formal communications, status reports, documentation
- [ ] Chat (Slack/Teams): Quick questions, informal updates, daily coordination
- [ ] Meetings: Collaborative decisions, complex discussions, alignment
- [ ] Project Dashboard: Real-time status, metrics, self-service information
- [ ] Documents: Detailed specifications, plans, reference materials

- [ ] Communicate deployment status

- [ ] Coordinate go/no-go communications

- [ ] Share change calendar and upcoming changes
- [ ] Report on change success rates and issues

- [ ] Communicate technical blockers

- [ ] Escalate risks and issues

- [ ] Share test plans and coverage

## Communication Effectiveness Review

### Monthly Review (Project Manager)
- [ ] Review communication feedback from stakeholders
- [ ] Assess if stakeholders have needed information
- [ ] Identify communication gaps or inefficiencies
- [ ] Update communication plan as needed
- [ ] Recognize effective communication practices

### Retrospective Questions
- Did stakeholders have the information they needed when they needed it?
- Were there any surprises that could have been communicated earlier?
- Which communication channels worked best?
- What communication improvements should we make?

## Emergency Communication Procedures

### Critical Issues Requiring Immediate Attention
1. **Identify**: Issue is business-impacting or blocks critical path
2. **Notify**: Immediately contact Project Manager and affected stakeholders
3. **Channel**: Use phone/direct message for urgent issues (not just email)
4. **Escalate**: Follow escalation matrix if no response within defined timeframe
5. **Document**: Log issue and communications in project tracking system
6. **Follow-up**: Provide resolution update to all notified parties

### Communication During Incidents
1. **Initial Alert**: Immediate notification of incident and severity
2. **Status Updates**: Every 30-60 minutes during active incident response
3. **Stakeholder Channels**: Use pre-defined incident communication channels
4. **Resolution Notice**: Confirm incident resolved and systems stable
5. **Post-Incident**: Schedule and communicate post-incident review findings
